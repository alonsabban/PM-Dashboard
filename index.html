Sync Trigger Rules

Aha → Jira triggers
  • Feature promoted to "Ready for Dev" status
  • Feature manually sent via integration button
  • Release date set on a feature
  • Feature assigned to a release in current sprint
Jira → Aha triggers
  • Status change on a linked Jira issue
  • Story points updated in Jira
  • Comment added to linked issue
  • Issue resolved / closed

Common Sync Issues & Fixes

Feature not appearing in Jira after push

1. Check Aha integration logs: Settings → Integrations → Jira → Log

2. Verify the target Jira project key matches in integration config

3. Ensure the Aha user has create issue permissions in Jira

4. Confirm the Jira issue type (Story/Feature) exists in the project

Status not syncing back from Jira

1. Check the Jira webhook is still registered: Jira → Settings → System → Webhooks

2. Verify Aha webhook URL is accessible from Jira (firewall / IP allowlist)

3. Confirm status workflow mapping is complete — unmapped statuses are silently ignored

Duplicate Jira issues created

1. This typically happens when the Jira issue key is missing from the Aha feature's external link field

2. Manually link the Aha feature to the existing Jira issue via the Integrations tab on the feature record

3. Enable duplicate detection in Aha integration settings if available

Rich text / attachments not syncing

1. Aha converts HTML to Jira wiki markup — complex tables may not render perfectly

2. Attachments require the Aha integration to have attachment sync enabled

3. Images embedded in descriptions are converted to external links

RICE Reference Guide

RICE RangePriorityAction
> 500Must doSchedule immediately
200 – 500HighNext 1-2 sprints
50 – 200MediumBacklog refinement
< 50LowRevisit quarterly

Reach: Users affected per period (month recommended)

Impact: Use 3×/2×/1×/0.5×/0.25× scale only (keeps scores comparable)

Confidence: 100% = hard data · 80% = some data · 50% = gut feel

Effort: Total work in person-months across the whole team

Capacity Tips

Rule of thumb: Start with 70% focus factor. Adjust down if the team has many meetings or concurrent projects.
Buffer: Reserve 10–20% of capacity for unplanned work / bugs.
Velocity vs. Capacity: Use actual team velocity from past 3 sprints as a sanity check against this number.
PTO: Subtract 1 member's capacity for every 2 days PTO in the sprint.
